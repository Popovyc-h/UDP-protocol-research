# UDP-protocol-research

## Завдання 1: Дослідження активних UDP з'єднань

### 1. Стани UDP-сокетів
Стани `LISTENING` або `ESTABLISHED` відсутні, оскільки UDP не встановлює з'єднання. Тому колонка зі станами залишається пустою.

### 2. Використані порти в системі

| Порт | Сервіс |
|------|--------|
| 123 | W32Time |
| 5050 | CDPSvc |
| 5353, 5355, 55472 | Dnscache |
| 1900, 54712, 54711 | SSDPSRV |
| 49664 | iphlpsvc |

### 3. Використання UDP браузерами
*   **HTTP/1, HTTP/2:** Майже не використовують UDP, оскільки вимагають надійної доставки (сторінка має завантажитись повністю і без помилок).
*   **HTTP/3:** Активно використовує UDP через протокол QUIC.

### Чому DNS використовує переважно UDP? В яких випадках він переходить на TCP?
* **DNS використовує переважно UDP тому, що запит і відповідь дуже маленькі і швидкі, а TCP використовує handshake так це надійно але це зайвий час. Перехід на TCP відбувається тоді коли DNS відповідь дуже велика тобто перевищує розмір пакету 512 байтів тоді вже працює TCP.**

### Як працює DHCP, якщо у клієнта ще немає IP-адреси?
* **DHCP працює по таким принципам він використовує (Broadcast - широкомовлення) 
Broadcast надсилає пакет інформації  який отримують і обробляють абсолютно всі пристрої у цій локальній мережі. Але відповідає на цей пакет тільки один пристрій тобто ваш роутер який реагує на цей запит належним чином і пропонує пристрою вільну IP-адресу.**

### Що таке QUIC і яке відношення він має до UDP?
* **QUIC - це сучасний мережевий протокол передачі даних, створений компанією Google для заміни застарілої зв'язки TCP + TLS, який робить інтернет-з'єднання значно швидшим та безпечнішим. QUIC побудований «поверх» UDP і використовує його як свій фундамент. Чому UDP а не TCP тому що UDP максимально простий і надшвидкий, але при цьому QUIC сам додає надійність і контроль доставки, яких немає у звичайному UDP**

## Завдання 3: Експерименти з надсиланням UDP пакетів

 ### Що станеться, якщо ви надішлете UDP пакет на порт, який ніхто не слухає? Чи отримає відправник якусь помилку?
 * **Якщо надіслати пакет на порт який ніхто не слухає то відправник отримає помилку (Ncat: An existing connection was forcibly closed by the remote host. .)**

### Чи зберігається порядок повідомлень, якщо надіслати 10 пакетів поспіль через nc?
* **Ні порядок не зберігається. UDP не гарантує порядок — кожен пакет іде своїм шляхом через мережу і може прийти в будь-якому порядку. Але є нюанс, якщовідправляєш на localhost (127.0.0.1) — порядок скоріш за все збережеться, бо пакети не йдуть через реальну мережу.**
